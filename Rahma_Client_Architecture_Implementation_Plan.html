Client Review Document

Rahma AI Guardian App Architecture and Implementation Plan

A practical Phase 1 and Phase 2 delivery plan covering system architecture, data flow, UX flow, server architecture, open technical options, recommended decisions, and launch readiness.

Prepared: 22 May 2026 Phase 1: Elderly and special-care pilot Phase 2: Children, geofencing, pharmacy, languages

1. Executive Summary

Rahma is planned as a one-touch emergency and care-assistance platform for vulnerable users. Phase 1 should prove the emergency-care foundation for elderly and special-care users. Phase 2 should expand the platform into children, geofencing, pharmacy communication, broader language support, and selected wearable integration.

Recommended build modelNative iOS and Android apps, reusable backend services, receiver dashboard, and client-approved UAE-hosted infrastructure.
Recommended hostingAWS UAE Region with a UAE-hosted foundation-model layer. Custom on-premises hosting should be treated as a separate proposal.
Recommended AI postureAI supports language and intent context. The one-touch activation itself starts the safety workflow.
We will meet any compliance, audit, data handling, or security requirement specifically confirmed by the client or authority stakeholders. Until those requirements are formally shared, the architecture will keep controls configurable and evidence-ready.

2. Confirmed Scope

Phase 1: Elderly and Special-Care Emergency Pilot

  • One-touch emergency activation with press-and-hold protection.
  • Voice and visual yes/no interaction running in parallel.
  • Arabic and English support.
  • Emergency event creation with timestamp, location, profile summary, and status.
  • Caregiver/family notification through push and SMS fallback.
  • Receiver dashboard for authority/support/family users.
  • Hospital finder using an approved facility directory.
  • Medication reminders for the user.
  • Offline and weak-network fallback using local data, last known location, SMS, and later sync.
  • Audit trail for emergency events, receiver actions, notification status, and support handling.

Phase 2: Children, Geofencing, Pharmacy, Scale, and Languages

  • Child profile module with parent/guardian access.
  • School or authorized receiver workflows.
  • Safe-zone/geofencing setup and alerts.
  • Family member messaging and escalation.
  • Pharmacy communication for prescription/dosage supply workflows.
  • Medication reminder expansion with missed-dose escalation.
  • Multilingual expansion beyond Arabic and English.
  • Wearable integration as a controlled workstream, depending on selected devices and APIs.

Our recommendation is UAE Pass as the primary route where approved, assisted onboarding as the elderly-friendly fallback, mobile OTP plus relationship approval for caregivers, and client SSO for staff/dashboard users.

5. How the App Works: Activation to Emergency Handoff

Rahma should be explainable to the client as a simple emergency workflow. The user presses and holds the main Rahma button for 2 seconds, the app starts the emergency workflow, captures location and minimal profile context, alerts caregivers/receivers, and then dispatches or hands off through the approved route.

  1. Activate Rahma: User presses and holds the main button for 2 seconds.
  2. Confirm context: App asks one simple voice/visual prompt such as Are you hurt? or Do you need help?
  3. Capture location and profile: App captures GPS/network location or falls back to last-known location with timestamp and confidence.
  4. Create emergency event: Backend creates a timestamped emergency event with activation source, user reference, location quality, and workflow status.
  5. Alert caregivers and receivers: Push/SMS alerts and dashboard events start immediately, with a normal-connectivity target of under 10 seconds.
  6. Dispatch or handoff: If a direct approved authority API exists, the emergency packet is sent through that route. If not, Phase 1 uses receiver dashboard, call-center/manual handoff, or simulator.
  7. Track and close: Receivers acknowledge, escalate, mark false alarm, or close the event. Every action is audited.
Recommended authority dispatch position: use the client-confirmed authority or emergency API route wherever it is available. Build the emergency packet, receiver dashboard, and audit trail as the shared foundation for every handoff route, while keeping approved dashboard, call-center, manual handoff, or simulator options as operational fallback paths during pilot validation.

9. Recommended Server Architecture

Preferred Option: AWS UAE Region with UAE-Hosted Foundation Model

This is the recommended option from our side for the Phase 1 pilot. It is faster for Phase 1, easier to operate, and better suited for scaling into Phase 2 than building a custom GPU environment from day one. A UAE-hosted foundation model can support higher-quality Arabic/English understanding while keeping sensitive patient data inside the approved hosting boundary.

Alternative: Custom On-Premises / Dedicated GPU Server

Use this only if the client requires a fully dedicated environment. It provides maximum infrastructure control but increases cost, procurement time, MLOps effort, security maintenance, and delivery risk.

Alternative: Hybrid

Use managed cloud for app services and a private AI server for selected sensitive model workloads. This can balance speed and control, but it adds integration complexity.

16. Final Recommendation

Rahma should proceed with a disciplined Phase 1 pilot focused on emergency reliability, accessibility, location capture, caregiver/receiver alerting, medication reminders, auditability, and operational readiness.

The most favorable technical path is native iOS and Android apps, AWS UAE Region with a UAE-hosted foundation-model layer, constrained on-device offline logic, SMS fallback, reusable backend services, and a receiver dashboard. This gives the client a realistic foundation while keeping Phase 2 expansion possible without rebuilding the platform.
